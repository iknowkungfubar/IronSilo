# Security Policy

At IronSilo, we take the security of your local AI fortress seriously. While our architecture is designed to be "local-first" and heavily sandboxed, we are committed to resolving vulnerabilities in our infrastructure, proxy, and agentic integrations as quickly as possible.

## Supported Versions

We currently provide security updates for the following versions of IronSilo:

| Version | Supported          | Notes |
| ------- | ------------------ | ----- |
| 2.x.x   | :white_check_mark: | Active Enterprise Orchestrator branch |
| < 2.0.0 | :x:                | Deprecated. Please upgrade to 2.x. |

## Reporting a Vulnerability

**Please do not report security vulnerabilities through public GitHub issues.**

If you believe you have found a security vulnerability in IronSilo, please report it to us privately by emailing **security@ironsilo.dev**. 

Please include the following information in your report:
* A description of the vulnerability and its impact.
* Steps to reproduce the issue.
* Your operating system and Docker configuration.
* Any relevant logs or output (please redact any personal API keys).

### What to Expect
1. We will acknowledge receipt of your vulnerability report within **48 hours**.
2. We will investigate the issue and determine if it is a valid vulnerability.
3. We will send you regular updates on our progress.
4. Once the issue is resolved, we will publish a security advisory and notify the community. We will credit you for the discovery (unless you prefer to remain anonymous).

## Scope and Threat Model

IronSilo is designed to run locally and provide agents (like IronClaw and Aider) with access to your local machine and vector databases. 

**The following are CONSIDERED vulnerabilities in our threat model:**
* **Container Escape:** An agent or external process breaking out of the Docker Sandbox (Khoj, Genesys, LLM Proxy) into the host OS.
* **WASM Sandbox Escape:** The IronClaw agent executing unauthorized system calls outside its WebAssembly runtime.
* **Authentication Bypass:** Bypassing the API Key requirements on the Proxy or MCP Servers to access databases or internal endpoints.
* **Cryptographic Failures:** Flaws in the AES-256-GCM implementation or key management system that allow data-at-rest to be decrypted without the master key.
* **Injection Attacks:** SQL injection in the PostgreSQL/pgvector database or Prompt Injection that bypasses proxy routing guardrails.

**The following are NOT considered vulnerabilities (Expected Behavior):**
* **Aider Modifying Local Files:** Aider is explicitly designed to read and write to your local workspace. This is the intended behavior of the Action Layer.
* **Local Network Visibility:** IronSilo binds to `127.0.0.1` and `host.docker.internal` by default. It is the user's responsibility to secure their local network and not expose these ports to the public internet unless properly reverse-proxied.
* **Third-Party Model Hallucinations:** Malicious or incorrect code generated by the LLM (e.g., Llama 3 or Qwen) you choose to run on your host machine.

## Best Practices for Users

To maintain the security of your IronSilo environment, we strongly recommend:
1. **Never commit your `.env` file.** Keep your `POSTGRES_PASSWORD` and `IRONSILO_API_KEY` entirely local.
2. **Review Aider Commits.** Always review the code diffs Aider generates before pushing them to production.
3. **Keep Docker Updated.** Ensure your container engine (Docker/Podman) is up-to-date to benefit from the latest cgroup and namespace security patches.
