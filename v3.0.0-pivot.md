Here is the comprehensive research brief and bulletproof build plan for the IronSilo Redesign.

1. Component Research: From-Scratch vs. Fused Stack
Given your strict requirement for Free and Open Source Software (FOSS) and the desire for a drop-in stack, a Fused Stack is highly recommended over building from scratch. Rebuilding RAG, Vector databases, and LLM serving from scratch introduces years of technical debt.

Here is the optimal FOSS stack to fuse:

A. The API Gateway / Single Port Router (New)

Recommendation: Traefik [^1] or Caddy.

Why: Traefik dynamically reads Docker labels. You expose only port 80/443 on the host. Traefik routes /api/llm to the LLMLingua Proxy, /khoj to the RAG UI, and /agent to your PDA.

B. The Bare-Metal Coder

Recommendation: OpenCode (per your preference).

Why: OpenCode operates securely in the host terminal, mapping the local workspace AST, and acts as the sole bridge between your local file system and the isolated Silo.

C. The Containerized PDA / Automation Agent (Replacing IronClaw)

Recommendation: Containerize OpenHands (formerly OpenDevin) or utilize your existing CDP Browser Swarm (swarm/harness_worker.py) wrapped in a lightweight FastAPI UI.

Why: IronClaw (WASM) on the host violates the "Silo" principle. A containerized OpenHands instance can browse the web privately (via SearXNG), read Khoj, and write to Genesys memory, all entirely inside the Docker network.

D. Containerized Inference (Lemonade / Local Models)

Recommendation: Ollama (for maximum cross-platform drop-in compatibility) or a custom Lemonade container [^2].

Why: Containerizing the inference engine brings the heavy lifting inside the Silo, standardizing the API endpoint for the proxy.

2. Roadblock & 'Gotcha' Analysis
Before executing, you must account for these hidden hurdles in the "True Silo" approach:

Gotcha 1: GPU Passthrough Fragmentation: Containerizing the LLM (Lemonade/Ollama) breaks the "cross-platform drop-in" promise if not handled carefully. Windows requires WSL2 GPU flags, Linux/NVIDIA requires the nvidia-container-toolkit, and Linux/AMD requires passing /dev/kfd and /dev/dri to the container.

Gotcha 2: The 4GB RAM Cap Paradox: Your current docker-compose.yml strictly caps the backend to ~4GB of RAM. If you move the LLM inference into the container stack, you must separate the Docker Compose resource limits. The LLM container needs unbounded (or highly generous) access to system RAM/VRAM, while the proxy/RAG/memory keep the strict 4GB cap.

Gotcha 3: Single-Port Path Stripping: RAG UIs (like Khoj) often expect to be hosted at the root domain (/). Routing them behind a single port via Traefik (e.g., localhost/khoj) requires careful rewrite middleware, otherwise, CSS/JS assets will 404.

Gotcha 4: Model Output Quality: You mentioned struggling with local model performance. Local 7B/8B models often fail at complex agentic tool-calling unless strictly formatted. You will need to rely heavily on your LLMLingua proxy to inject strict System Prompts and grammar constraints before requests hit the containerized model.

3. The Recommended Architecture: "The True Silo"
You should take the Fused Stack approach. You already have incredible custom glue code (Genesys causal memory, LLMLingua proxy, Swarm CDP). Do not throw this away. Refactor the deployment topology.

Why this route? It perfectly achieves your "Stack in a Box" goal without reinventing the wheel.

The Architecture Topology:

Plaintext
[ BARE METAL HOST ]
      │
      ├── Source Code Workspace
      └── OpenCode CLI (Targeting http://localhost:8080/api/llm)
      │
[ DOCKER NETWORK PERIMETER ] ◄── ONLY PORT 8080 EXPOSED
      │
      ▼
[ TRAEFIK REVERSE PROXY ] (Handles Auth & Routing)
      ├── Route: /api/llm  ──► [ LLMLingua Proxy ] ──► [ Containerized Lemonade/Ollama ]
      ├── Route: /memory   ──► [ Genesys API ]     ──► [ pgvector ]
      ├── Route: /rag      ──► [ Khoj Web UI ]
      └── Route: /pda      ──► [ Containerized PDA / OpenHands ] ──► [ SearXNG / Browser Node ]
Security Bonus: The current audit (C01: No Auth) is instantly solved. You can place Traefik Basic Auth or an API-Key middleware globally at the single exposed port.

4. Phased Agentic Build Plan
Execute this roadmap using OpenCode.

Phase 1: Perimeter Consolidation (API Gateway)
Goal: Close all exposed ports and route everything through Traefik.

Action: Modify docker-compose.yml. Remove all ports: mappings from existing services (except the new Traefik service). Add Traefik labels to route /api, /rag, and /memory to their respective internal containers.

Phase 2: Inference Isolation
Goal: Containerize the LLM and optimize model output.

Action: Add an inference-engine service to docker-compose.yml (using an official Ollama image or custom Lemonade Dockerfile). Configure hardware passthrough (deploy.resources.reservations.devices). Update the Proxy configuration to point to http://inference-engine:8000 on the internal Docker network.

Phase 3: The PDA Migration
Goal: Move the Personal Digital Assistant off bare metal and into the Silo.

Action: Deprecate IronClaw. Integrate a containerized agent UI (or build a simple FastAPI frontend for your existing swarm/orchestrator.py) and expose it via Traefik at /pda.

Phase 4: Bare-Metal OpenCode Integration
Goal: Connect OpenCode securely to the Silo.

Action: Generate a master API key. Configure the local OpenCode environment variables to point to http://localhost:8080/api/llm with the required authorization headers.
